ty enforces Python's type system through a comprehensive set of rules. Each rule detects a specific category of type error or potential issue in your code.

## Understanding rules

Each rule has:
- A unique identifier (e.g., `invalid-assignment`)
- A default severity level (`error`, `warn`, or `ignore`)
- A description of what it checks
- Examples of problematic code

You can configure individual rules in your `pyproject.toml`. See the [configuration guide](/guides/configuration) for details.

## Rule categories

Rules are organized by the type of issue they detect:

- **Type errors**: Core type system violations
- **Class and inheritance**: Issues with class definitions and inheritance
- **Functions and callables**: Problems with function signatures and calls
- **Dataclasses**: Dataclass-specific validation
- **Annotations**: Issues with type annotations
- **Type constructs**: Problems with type aliases, generics, and protocols
- **Code quality**: Best practices and potential bugs

## All rules

<Accordion title="abstract-method-in-final-class">

**Default level**: `error`

**What it does**

Checks for `@final` classes that have unimplemented abstract methods.

**Why is this bad?**

A class decorated with `@final` cannot be subclassed. If such a class has abstract methods that are not implemented, the class can never be properly instantiated, as the abstract methods can never be implemented (since subclassing is prohibited).

**Example**

```python
from abc import ABC, abstractmethod
from typing import final

class Base(ABC):
    @abstractmethod
    def method(self) -> int: ...

@final
class Derived(Base):  # Error: `Derived` does not implement `method`
    pass
```

</Accordion>

<Accordion title="ambiguous-protocol-member">

**Default level**: `warn`

**What it does**

Checks for protocol classes with members that will lead to ambiguous interfaces.

**Why is this bad?**

Assigning to an undeclared variable in a protocol class leads to an ambiguous interface which may lead to the type checker inferring unexpected things. It's recommended to ensure that all members of a protocol class are explicitly declared.

**Example**

```python
from typing import Protocol

class BaseProto(Protocol):
    a: int                               # fine (explicitly declared as `int`)
    def method_member(self) -> int: ...  # fine: a method definition using `def` is considered a declaration
    c = "some variable"                  # error: no explicit declaration, leading to ambiguity
    b = method_member                    # error: no explicit declaration, leading to ambiguity
```

</Accordion>

<Accordion title="assert-type-unspellable-subtype">

**Default level**: `error`

**What it does**

Checks for `assert_type()` calls where the actual type is an unspellable subtype of the asserted type.

**Why is this bad?**

`assert_type()` is intended to ensure that the inferred type of a value is exactly the same as the asserted type. But in some situations, ty has nonstandard extensions to the type system that allow it to infer more precise types than can be expressed in user annotations.

**Example**

```python
def _(x: int):
    assert_type(x, int)  # fine
    if x:
        assert_type(x, int)  # error: [assert-type-unspellable-subtype]
                             # the actual type is `int & ~AlwaysFalsy`,
                             # which excludes types like `Literal[0]`
```

</Accordion>

<Accordion title="byte-string-type-annotation">

**Default level**: `error`

**What it does**

Checks for byte-strings in type annotation positions.

**Why is this bad?**

Static analysis tools like ty can't analyze type annotations that use byte-string notation.

**Example**

```python
def test(): -> b"int":
    ...
```

</Accordion>

<Accordion title="call-abstract-method">

**Default level**: `error` (Preview)

**What it does**

Checks for calls to abstract `@classmethod`s or `@staticmethod`s with "trivial bodies" when accessed on the class object itself.

**Why is this bad?**

An abstract method with a trivial body has no concrete implementation to execute, so calling such a method directly on the class will probably not have the desired effect.

**Example**

```python
from abc import ABC, abstractmethod

class Foo(ABC):
    @classmethod
    @abstractmethod
    def method(cls) -> int: ...

Foo.method()  # Error: cannot call abstract classmethod
```

</Accordion>

<Accordion title="call-non-callable">

**Default level**: `error`

**What it does**

Checks for calls to non-callable objects.

**Why is this bad?**

Calling a non-callable object will raise a `TypeError` at runtime.

**Example**

```python
4()  # TypeError: 'int' object is not callable
```

</Accordion>

<Accordion title="call-top-callable">

**Default level**: `error`

**What it does**

Checks for calls to objects typed as `Top[Callable[..., T]]` (the infinite union of all callable types with return type `T`).

**Why is this bad?**

When an object is narrowed to `Top[Callable[..., object]]`, we know the object is callable, but we don't know its precise signature. No specific set of arguments can be guaranteed to be valid.

**Example**

```python
def f(x: object):
    if callable(x):
        x()  # error: We know `x` is callable, but not what arguments it accepts
```

</Accordion>

<Accordion title="conflicting-argument-forms">

**Default level**: `error`

**What it does**

Checks whether an argument is used as both a value and a type form in a call.

**Why is this bad?**

Such calls have confusing semantics and often indicate a logic error.

**Example**

```python
from typing import reveal_type
from ty_extensions import is_singleton

if flag:
    f = repr  # Expects a value
else:
    f = is_singleton  # Expects a type form

f(int)  # error
```

</Accordion>

<Accordion title="conflicting-declarations">

**Default level**: `error`

**What it does**

Checks whether a variable has been declared as two conflicting types.

**Why is this bad**

A variable with two conflicting declarations likely indicates a mistake. Moreover, it could lead to incorrect or ill-defined type inference for other code that relies on these variables.

**Example**

```python
if b:
    a: int
else:
    a: str

a = 1
```

</Accordion>

<Accordion title="conflicting-metaclass">

**Default level**: `error`

**What it does**

Checks for class definitions where the metaclass of the class being created would not be a subclass of the metaclasses of all the class's bases.

**Why is it bad?**

Such a class definition raises a `TypeError` at runtime.

**Example**

```python
class M1(type): ...
class M2(type): ...
class A(metaclass=M1): ...
class B(metaclass=M2): ...

# TypeError: metaclass conflict
class C(A, B): ...
```

</Accordion>

<Accordion title="cyclic-class-definition">

**Default level**: `error`

**What it does**

Checks for class definitions in stub files that inherit (directly or indirectly) from themselves.

**Why is it bad?**

Although forward references are natively supported in stub files, inheritance cycles are still disallowed, as it is impossible to resolve a consistent method resolution order for a class that inherits from itself.

**Example**

```python
# foo.pyi
class A(B): ...
class B(A): ...
```

</Accordion>

<Accordion title="cyclic-type-alias-definition">

**Default level**: `error`

**What it does**

Checks for type alias definitions that (directly or mutually) refer to themselves.

**Why is it bad?**

Although it is permitted to define a recursive type alias, it is not meaningful to have a type alias whose expansion can only result in itself, and is therefore not allowed.

**Example**

```python
type Itself = Itself

type A = B
type B = A
```

</Accordion>

<Accordion title="dataclass-field-order">

**Default level**: `error` (Preview)

**What it does**

Checks for dataclass definitions where required fields are defined after fields with default values.

**Why is this bad?**

In dataclasses, all required fields (fields without default values) must be defined before fields with default values. This is a Python requirement that will raise a `TypeError` at runtime if violated.

**Example**

```python
from dataclasses import dataclass

@dataclass
class Example:
    x: int = 1    # Field with default value
    y: str        # Error: Required field after field with default
```

</Accordion>

**Why is this bad?**

Deprecated items should no longer be used.

**Example**

</Accordion>

<Accordion title="division-by-zero">

**Default level**: `ignore`

**What it does**

It detects division by zero.

**Why is this bad?**

**Rule status**

**Example**

```python

```

</Accordion>

<Accordion title="duplicate-base">

**Default level**: `error`

**What it does**

Checks for class definitions with duplicate bases.

**Why is this bad?**

Class definitions with duplicate bases raise `TypeError` at runtime.

**Example**

```python

```

</Accordion>

<Accordion title="duplicate-kw-only">

**Default level**: `error`

**What it does**

Checks for dataclass definitions with more than one field annotated with `KW_ONLY`.

**Why is this bad?**

`dataclasses.KW_ONLY` is a special marker used to emulate the `*` syntax in normal signatures. It can only be used once per dataclass.

**Example**

```python
from dataclasses import dataclass, KW_ONLY

@dataclass
class A:  # Crash at runtime
    b: int
    _1: KW_ONLY
    c: str
    _2: KW_ONLY
    d: bytes
```

</Accordion>

<Accordion title="empty-body">

**Default level**: `error`

**What it does**

Detects functions with empty bodies that have a non-`None` return type annotation.

**Why is this bad?**

A function with an empty body (containing only `...`, `pass`, or a docstring) will implicitly return `None` at runtime. Returning `None` when the return type is non-`None` is unsound, and will lead to ty inferring incorrect types elsewhere.

**Example**

```python
def foo() -> int: ...  # error: [empty-body]

def bar() -> str:
    """A function that does nothing."""
    pass  # error: [empty-body]
```

</Accordion>

<Accordion title="escape-character-in-forward-annotation">

**Default level**: `error`

**What it does**

Checks for forward annotations that contain escape characters.

**Why is this bad?**

Static analysis tools like ty can't analyze type annotations that contain escape characters.

**Example**


</Accordion>

<Accordion title="final-on-non-method">

**Default level**: `error`

**What it does**

Checks for `@final` decorators applied to non-method functions.

**Why is this bad?**

The `@final` decorator is only meaningful on methods and classes. Applying it to a module-level function or a nested function has no effect and is likely a mistake.

**Example**

```python
from typing import final

# Error: @final is not allowed on non-method functions
@final
def my_function() -> int:
    return 0
```

</Accordion>

<Accordion title="final-without-value">

**Default level**: `error`

**What it does**

Checks for `Final` symbols that are declared without a value and are never assigned a value in their scope.

**Why is this bad?**

A `Final` symbol must be initialized with a value at the time of declaration or in a subsequent assignment.

**Example**

```python
from typing import Final

# Error: `Final` symbol without a value
MY_CONSTANT: Final[int]

# OK: `Final` symbol with a value
MY_CONSTANT: Final[int] = 1
```

</Accordion>

<Accordion title="invalid-argument-type">

**Default level**: `error`

**What it does**

Checks for function calls where an argument's type doesn't match the parameter's type.

**Why is this bad?**

Passing arguments of the wrong type violates the function's contract and may cause runtime errors.

**Example**

```python
def greet(name: str) -> None:
    print(f"Hello, {name}!")

greet(42)  # error: expected str, got int
```

</Accordion>

<Accordion title="invalid-assignment">

**Default level**: `error`

**What it does**

Checks for assignments where the type of the value is not assignable to the type of the assignee.

**Why is this bad?**

Such assignments break the rules of the type system and weaken a type checker's ability to accurately reason about your code.

**Example**

```python
a: int = ''  # error: cannot assign str to int
```

</Accordion>

<Accordion title="invalid-attribute-access">

**Default level**: `error`

**What it does**

Checks for assignments to class variables from instances and assignments to instance variables from its class.

**Why is this bad?**

Incorrect assignments break the rules of the type system and weaken a type checker's ability to accurately reason about your code.

**Example**

```python
class C:
    class_var: ClassVar[int] = 1
    instance_var: int

C.class_var = 3  # okay
C().class_var = 3  # error: Cannot assign to class variable

C().instance_var = 3  # okay
C.instance_var = 3  # error: Cannot assign to instance variable
```

</Accordion>

<Accordion title="invalid-await">

**Example**

**Example**

<Accordion title="useless-overload-body">

**Default level**: `warn`

**What it does**

Checks for various `@overload`-decorated functions that have non-stub bodies.

**Why is this bad?**

Functions decorated with `@overload` are ignored at runtime; they are overridden by the implementation function that follows the series of overloads. While it is not illegal to provide a body for an `@overload`-decorated function, it may indicate a misunderstanding of how the `@overload` decorator works.

**Example**

```python
from typing import overload

@overload
def foo(x: int) -> int:
    return x + 1  # will never be executed

@overload
def foo(x: str) -> str:
    return "Oh no, got a string"  # will never be executed

def foo(x: int | str) -> int | str:
    raise Exception("unexpected type encountered")
```

</Accordion>

<Accordion title="zero-stepsize-in-slice">

**Default level**: `error`

**What it does**

Checks for step size 0 in slices.

**Why is this bad?**

A slice with a step size of zero will raise a `ValueError` at runtime.

**Example**

```python
l = list(range(10))
l[1:10:0]  # ValueError: slice step cannot be zero
```

</Accordion>

## Configuring rules

You can configure individual rules in your `pyproject.toml`:

```toml
[tool.ty]
# Set global severity level
level = "error"

# Configure specific rules
[tool.ty.rules]
deprecated = "error"  # Treat deprecated warnings as errors
division-by-zero = "error"  # Enable division by zero checks
```

See the [configuration reference](/reference/configuration) for more details.
