title: Editor settings
description: Editor-specific settings and configuration for ty's language server

The editor settings supported by ty's language server, as well as settings specific to ty's VS Code extension.

## Language server settings

These settings are supported across all editors that use ty's language server.

<ParamField path="configuration" type="object" default="null">
  In-editor configuration of ty's settings.

  The inline settings always take precedence over the settings from configuration files, including the configuration specified with `configurationFile`.

  Consult the [configuration reference](/reference/configuration) for a list of all supported configuration options.

  <Accordion title="Example: VS Code">
    ```json
    {
      "ty.configuration": {
        "rules": {
          "unresolved-reference": "warn"
        }
      }
    }
    ```
  </Accordion>

  <Accordion title="Example: Neovim">
    ```lua
    -- Neovim >=0.11:
    vim.lsp.config('ty', {
      settings = {
        ty = {
          configuration = {
            rules = {
              ["unresolved-reference"] = "warn"
            }
          }
        },
      },
    })

    -- Neovim <0.11:
    require('lspconfig').ty.setup({
      settings = {
        ty = {
          configuration = {
            rules = {
              ["unresolved-reference"] = "warn"
            }
          }
        },
      },
    })
    ```
  </Accordion>

  <Accordion title="Example: Zed">
    ```json
    {
      "lsp": {
        "ty": {
          "settings": {
            "configuration": {
              "rules": {
                "unresolved-reference": "warn"
              }
            }
          }
        }
      }
    }
    ```
  </Accordion>
</ParamField>

<ParamField path="configurationFile" type="string" default="null">
  The path to a `ty.toml` configuration file.

  ty will use the specified configuration over any automatically discovered configuration. ty will expand a tilde `~` at the start of a string to the user's home directory, as well as variables like `$A` or `${A}`.

  <Info>While ty configuration can be included in a `pyproject.toml` file, it is not allowed in this context.</Info>

  <Accordion title="Example: VS Code">
    ```json
    {
      "ty.configurationFile": "./.config/ty.toml"
    }
    ```
  </Accordion>

  <Accordion title="Example: Neovim">
    ```lua
    -- Neovim >=0.11:
    vim.lsp.config('ty', {
      settings = {
        ty = {
          configurationFile = "./.config/ty.toml"
        },
      },
    })

    -- Neovim <0.11:
    require('lspconfig').ty.setup({
      settings = {
        ty = {
          configurationFile = "./.config/ty.toml"
        },
      },
    })
    ```
  </Accordion>

  <Accordion title="Example: Zed">
    ```json
    {
      "lsp": {
        "ty": {
          "settings": {
            "configurationFile": "./.config/ty.toml"
          }
        }
      }
    }
    ```
  </Accordion>
</ParamField>

<ParamField path="disableLanguageServices" type="boolean" default="false">
  Whether to disable the language services for the ty language server like code completion, hover, go to definition, etc.

  This is useful if you want to use ty exclusively for type checking and want to use another language server for features like code completion, hover, go to definition, etc.

  <Accordion title="Example: VS Code">

  </Accordion>

  <Accordion title="Example: Neovim">

    ```lua
      settings = {
        ty = {
          disableLanguageServices = true,

      },
    })

    -- Neovim <0.11:
    require('lspconfig').ty.setup({
      settings = {
        ty = {
    ```

</ParamField>

<ParamField path="diagnosticMode" type="'off' | 'workspace' | 'openFilesOnly'" default="'openFilesOnly'">
  Determines the scope of the diagnostics reported by the language server.

  Setting this to `off` is useful if you want to use ty exclusively for the language server features like code completion, hover, go to definition, etc.

  - `off`: Diagnostics are disabled
  - `openFilesOnly`: Diagnostics are reported only for files that are currently open in the editor
  - `workspace`: Diagnostics are reported for all files in the workspace

  <Accordion title="Example: VS Code">
    ```json
    {
      "ty.diagnosticMode": "workspace"
    }
    ```
  </Accordion>

  <Accordion title="Example: Neovim">
    ```lua
    -- Neovim >=0.11:
    vim.lsp.config('ty', {
      settings = {
        ty = {
          diagnosticMode = 'workspace',
        },
      },
    })

    -- Neovim <0.11:
    require('lspconfig').ty.setup({
      settings = {
        ty = {
          diagnosticMode = 'workspace',
        },
      },
    })
    ```
  </Accordion>

  <Accordion title="Example: Zed">
    ```json
    {
      "lsp": {
        "ty": {
          "settings": {
            "diagnosticMode": "workspace"
          }
        }
      }
    }
    ```
  </Accordion>
</ParamField>

## VS Code specific settings

The following settings are specific to ty's VS Code extension.

<ParamField path="importStrategy" type="'fromEnvironment' | 'useBundled'" default="'fromEnvironment'">
  Strategy for loading the `ty` executable.

  - `fromEnvironment` finds ty in the environment, falling back to the bundled version
  - `useBundled` uses the version bundled with the extension

  <Accordion title="Example">
    ```json
    {
      "ty.importStrategy": "useBundled"
    }
    ```
  </Accordion>
</ParamField>

<ParamField path="interpreter" type="string[]" default="[]">
  A list of paths to Python interpreters. Even though this is a list, only the first interpreter is used.

  The interpreter path is used to find the `ty` executable when `ty.importStrategy` is set to `fromEnvironment`.

  <Accordion title="Example">
    ```json
    {
      "ty.interpreter": ["/home/user/.local/bin/python"]
    }
    ```
  </Accordion>
</ParamField>

<ParamField path="path" type="string[]" default="[]">
  A list of path to `ty` executables.

  The extension uses the first executable that exists. This setting takes precedence over the `ty.importStrategy` setting.

  <Accordion title="Example">
    ```json
    {
      "ty.path": ["/home/user/.local/bin/ty"]
    }
    ```
  </Accordion>
</ParamField>

<ParamField path="trace.server" type="'off' | 'messages' | 'verbose'" default="'off'">
  The detail level at which messages between the language server and the editor (client) are logged.

  This setting is useful for debugging issues with the language server. Refer to the [troubleshooting guide](https://github.com/astral-sh/ty-vscode/blob/6cf16b4e87342a49f2bec1310a730cde8229e1d9/TROUBLESHOOTING.md) for more information.

  <Accordion title="Example">
    ```json
    {
      "ty.trace.server": "messages"
    }
    ```
  </Accordion>
</ParamField>

## Initialization options

The following settings are required when ty is initialized in an editor. These settings are static so changing them requires restarting the editor to take effect.

For VS Code users, these settings are defined in the `ty.*` namespace as usual, but for other editors, they would need to be provided in a separate field of the configuration that corresponds to the initialization options.

<ParamField path="logFile" type="string" default="null">
  Path to the file to which the language server writes its log messages.

  By default, ty writes log messages to stderr.

  <Accordion title="Example: VS Code">
    ```json
    {
      "ty.logFile": "/path/to/ty.log"
    }
    ```
  </Accordion>

  <Accordion title="Example: Neovim">
    ```lua
    -- Neovim >=0.11:
    vim.lsp.config('ty', {
      init_options = {
        logFile = '/path/to/ty.log',
      },
    })

    -- Neovim <0.11:
    require('lspconfig').ty.setup({
      init_options = {
        logFile = '/path/to/ty.log',
      },
    })
    ```
  </Accordion>

  <Accordion title="Example: Zed">
    ```json
    {
      "lsp": {
        "ty": {
          "initialization_options": {
            "logFile": "/path/to/ty.log"
          }
        }
      }
    }
    ```
  </Accordion>
</ParamField>

<ParamField path="logLevel" type="'trace' | 'debug' | 'info' | 'warn' | 'error'" default="'info'">
  The log level to use for the language server.

  <Accordion title="Example: VS Code">
